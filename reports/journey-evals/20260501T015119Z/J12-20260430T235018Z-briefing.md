**Missing Information That Would Have Helped**
- A clear statement of what the CLI can do while the dataset is provisioning.
- A concrete estimate or status for when `econ` might become queryable.
- A lightweight alternative, such as a metadata/profile summary, if the dataset is not ready.
- A focused follow-up question, such as which dimension the user cares about: trends, anomalies, growth, outliers, or data quality.
- A cost/scope note before launching any background work.

**Information That Should Be Removed Or De-emphasized**
- Internal tool names like `list_remote_datasets` and `inspect_remote_dataset` should be de-emphasized in the normal UX. They are implementation details, not user value.
- “Inspected remote dataset econ” should be replaced with a user-facing result summary or hidden behind a debug mode.
- The generic “Say ‘yes’” phrasing should be replaced with explicit actions or numbered options.

**Suggested UI/Output Changes**
- When the dataset is provisioning, show a dedicated blocking state: “`econ` is not queryable yet. I can wait, retry, or draft a briefing from metadata.”
- Replace the current yes/no ask with explicit choices, for example:
  - “Wait and analyze when ready”
  - “Give me a metadata briefing now”
  - “Help me pick a research direction”
- Add a short explanation of what “Describe run” means and whether it starts immediately or waits for provisioning.
- Show progress as a user-facing state, not just tool-call text, so the screen does not feel idle for 5 to 15 seconds.
- Prefer one concise question over a bundled two-action ask.
