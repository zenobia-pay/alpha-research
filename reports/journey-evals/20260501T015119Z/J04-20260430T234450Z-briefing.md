# J04 UX Briefing: Dataset Briefing

## Verdict

Partial

The CLI correctly avoided starting duplicate work on a busy dataset, but it did not satisfy the user's dataset-briefing intent. A normal user saw a run-lifecycle block, not a dataset briefing, and the next action was vague: "ask me to check that run instead" rather than automatically retrieving or summarizing the existing run.

## User Input Burden

Moderate.

The user's input was simple and appropriate: "Describe the econ dataset for me." The CLI required the user to understand that "econ dataset" maps to a remote dataset, that a dataset can be "busy" with a run, and that they must either open a dashboard URL or issue a follow-up request to check the running job. The user was not given a direct CLI command or an automatic continuation path.

## Correct Behavior Assessment

Expected behavior for this journey: start or return a dataset briefing scoped to inventory and documentation, ideally with fields, measures, time coverage, source coverage, row counts, limitations, and prominent artifact links.

Actual behavior:

- At 5 seconds, the CLI showed it was retrieving remote dataset inventory: `Calling list_remote_datasets` and `Found 7 remote datasets` (`snapshots/0001-5s.txt`; `events.jsonl` at 3199ms and 3850ms).
- By the final snapshot, it had called `describe_remote_dataset`, then reported that the dataset was already busy with run `486be6ce-969a-4d2b-b353-acc6c80f1cb4` (`snapshots/0002-final.txt`; `terminal.log`).
- It exited with code 0 after 7233ms (`metadata.json`), even though the user did not receive a dataset briefing.

Behavior choice: reporting the block was appropriate, but the CLI should have moved into "wait/retrieve/debug existing run" behavior or made a clearer choice available. For this canonical journey, stopping at the busy-run message is only partially correct.

## Confusing Moments, Ordered By Severity

1. **The requested dataset briefing never appears.**

   The final visible output contains no dataset name, schema, row count, time coverage, source coverage, limitations, or artifact links. It only reports a busy run (`snapshots/0002-final.txt`, `terminal.log`). A normal user asking "Describe the econ dataset for me" would not know anything more about the dataset.

2. **"Dataset is already busy" is a lifecycle concept exposed without enough context.**

   The message says the dataset is busy with a running run, but does not explain whether that run is producing the requested briefing, who started it, when it started, whether it is healthy, or whether waiting is recommended (`snapshots/0002-final.txt`). This is run lifecycle confusion rather than dataset confusion.

3. **The suggested next step is conversational but not actionable in non-interactive prompt mode.**

   The CLI says "ask me to check that run instead of starting another one" (`terminal.log`). In `research --prompt` mode, the process then exits successfully. The user is left to invent the next command. A better output would include a direct command such as `research debug run 486be6ce-969a-4d2b-b353-acc6c80f1cb4` or automatically check the run.

4. **The busy-run message is duplicated.**

   The same long sentence appears once as a tool/status line and once as the final assistant response (`snapshots/0002-final.txt`, `terminal.log`). The duplication makes the final state feel noisier without adding information.

5. **The dashboard URL dominates the output and wraps awkwardly.**

   In the final 120-column snapshot, the dashboard link wraps across multiple lines and interrupts readability (`snapshots/0002-final.txt`). The long URL is useful, but it should be labeled and separated from the explanatory sentence.

6. **The first snapshot is blank.**

   `snapshots/0000-start.txt` contains only whitespace. For a short command, this is not severe, but a visible "starting" state or echoed prompt would reduce uncertainty during capture.

## Confusion Categories

Product confusion:

- The CLI does not make clear what it is doing to satisfy "Describe the econ dataset." It shows internal-ish tool calls, but not a user-facing plan or selected dataset.
- It exits with success despite not completing the briefing.

Dataset confusion:

- The user never sees which "econ dataset" was selected.
- No dataset facts are shown, so the user cannot judge dataset fitness.

Auth confusion:

- None visible. There are no login prompts, token errors, or permission messages.

Run lifecycle confusion:

- The central blocker is a running run on the dataset.
- The CLI does not show run age, status detail, owner/session, expected wait behavior, or whether results/artifacts already exist.
- The recommended next action is not concrete enough for prompt mode.

Terminal/UI readability problems:

- Duplicate busy-run text.
- Long dashboard URL wraps poorly.
- Tool-call bullets are sparse and do not show the selected dataset or why `describe_remote_dataset` was chosen.

## Density Judgment

Too sparse on useful product and dataset information; slightly too dense in repeated lifecycle/link text.

The amount of output is short, but the wrong information is emphasized. The final screen should prioritize the run state, selected dataset, next CLI action, and any available artifact/status summary. Instead, it repeats a long blocker sentence and a dashboard URL.

## Missing Information That Would Have Helped

- Selected dataset identifier and display name for "econ dataset."
- Whether the running run is the same task type as a dataset briefing.
- Current run status, age, last event, and whether artifacts already exist.
- A direct CLI follow-up command to inspect or wait for the run.
- A short explanation of why a new briefing cannot start while another run is active.
- Any partial dataset profile available without starting a run: fields, row count, source, time coverage, or existing documentation.

## Information To Remove Or De-emphasize

- Remove the duplicated busy-run paragraph. Show it once as the final user-facing result.
- De-emphasize raw tool-call wording once the final answer is shown, or collapse it behind concise progress states.
- Shorten or label the dashboard URL instead of embedding it inside a long sentence.

## Suggested UI/Output Changes

1. When `describe_remote_dataset` hits a busy run, automatically inspect that run if possible and show:

   ```text
   The econ dataset is currently being processed by run 486be6ce...1cb4.
   Status: running for 12m, last event 30s ago.
   I can show the run status now; no new dataset job was started.
   ```

2. In non-interactive `--prompt` mode, include concrete next commands:

   ```text
   Check it: research debug run 486be6ce-969a-4d2b-b353-acc6c80f1cb4
   Dashboard: https://...
   ```

3. If a lightweight dataset description is available without a run, show it despite the busy state:

   ```text
   Dataset: econ
   Rows: ...
   Time coverage: ...
   Measures: ...
   Limitation: full generated briefing is blocked by an active run.
   ```

4. Replace duplicated status/final text with a single final block:

   ```text
   Blocked: dataset is busy
   Dataset: econ
   Active run: 486be6ce-969a-4d2b-b353-acc6c80f1cb4 (running)
   Next: research debug run 486be6ce-969a-4d2b-b353-acc6c80f1cb4
   Dashboard: ...
   ```

5. Label progress messages with user-facing objects:

   ```text
   Looking up remote datasets...
   Found 7 datasets.
   Selected econ. Checking whether a briefing can run...
   ```
