**Verdict: Partial**

**User Input Burden**
- Low on the user side: they asked one simple prompt, `Describe the econ dataset for me.`
- High on the interpretation side: the CLI made them parse a busy-state block, a raw run id, and a long dashboard URL instead of getting the requested briefing.

**Correct Behavior Assessment**
- The lifecycle decision was directionally correct only in one respect: it did not start a duplicate run once it detected the dataset was busy.
- The user-facing behavior was not correct for this journey. The CLI said it was “starting dataset briefing” and only afterward reported the block, which makes the flow feel like a failed attempt rather than a deliberate, explained stop.
- For a briefing request, the right end state would be either:
  - a completed dataset briefing, or
  - an immediate, clear “existing run is already handling this” handoff with a friendlier continuation path.

**Confusing Moments, Ordered by Severity**
1. The user never got the dataset briefing they asked for.
   - The run ends after a block message; there is no inventory, profile, coverage, row counts, or limitations.
   - Evidence: [`0002-final.svg`](file:///Users/ryanprendergast/Documents/Zenobia%20Pay/alpha-datasets/.tmp/journey-runs-postfix/J04/20260501T184816Z/screenshots/0002-final.svg), [`terminal.log`](file:///Users/ryanprendergast/Documents/Zenobia%20Pay/alpha-datasets/.tmp/journey-runs-postfix/J04/20260501T184816Z/terminal.log)

2. The CLI framed the action as if it was proceeding, then reversed course.
   - Visible sequence: “Checking remote datasets…” → “Found 7 remote datasets.” → “Starting dataset briefing for econ…” → “Blocked: dataset is already busy.”
   - That creates a false start and makes the block feel like a failure, not a normal lifecycle state.
   - Evidence: [`0001-5s.svg`](file:///Users/ryanprendergast/Documents/Zenobia%20Pay/alpha-datasets/.tmp/journey-runs-postfix/J04/20260501T184816Z/screenshots/0001-5s.svg), [`0002-final.svg`](file:///Users/ryanprendergast/Documents/Zenobia%20Pay/alpha-datasets/.tmp/journey-runs-postfix/J04/20260501T184816Z/screenshots/0002-final.svg), [`events.jsonl`](file:///Users/ryanprendergast/Documents/Zenobia%20Pay/alpha-datasets/.tmp/journey-runs-postfix/J04/20260501T184816Z/events.jsonl)

   - “Active run: 68d1590c-01fd-428d-bc17-4e0685c2769b” and “research debug run …” are operational details, not a user-friendly explanation.

5. The start state is visually empty.
   - The first screenshot is basically blank, so the user gets no immediate sense of progress or mode.
   - Evidence: [`0000-start.svg`](file:///Users/ryanprendergast/Documents/Zenobia%20Pay/alpha-datasets/.tmp/journey-runs-postfix/J04/20260501T184816Z/screenshots/0000-start.svg)

**Missing Information That Would Have Helped**
- A plain-English explanation of why the briefing could not start.
- Whether the active run is the user’s own previous run or someone else’s.
- Whether the CLI will retry automatically or requires manual intervention.
- A direct “open existing run” action or shortcut instead of a debug command.
- A short summary of what the existing run is doing, so the user can decide whether to wait.
- If `econ` matched multiple datasets, a short disambiguation list instead of only “Found 7 remote datasets.”

**Information That Should Be Removed or De-Emphasized**
- The `research debug run <id>` instruction in the normal UX path.
- The full dashboard URL as primary output; it should be secondary or hidden behind a short link/action.
- “No new run was started.” because it is redundant once the block is shown.
- The raw run id should be de-emphasized unless the user explicitly asked for diagnostics.

**Suggested UI / Output Changes**
- Replace the current sequence with a single, explicit busy-state message when a dataset is occupied.
- Show the user:
  - what dataset they asked for,
  - that it is already being handled,
  - what the active run is doing,
  - and the next best action.
- Use a user-facing CTA like “Open existing run” or “Wait for this run to finish” instead of a debug command.
- If a briefing cannot begin, do not say “Starting dataset briefing…” first.
- Make dataset briefing mode explicit, but only after the CLI knows it can actually proceed.
- Compress URLs and operational IDs into secondary details so the core status stays readable.
