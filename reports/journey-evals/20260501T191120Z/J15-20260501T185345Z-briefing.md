**Correct Behavior Assessment**

**Missing Information That Would Have Helped**
- A plain-English summary of the state, such as “The run is still starting” or “This looks normal so far.”
- Whether `booting` is a normal startup phase or a sign of trouble.
- A simple rule for when to wait versus when to debug.
- A sense of whether the current update age is healthy, borderline, or stale.
- Any mapping from the raw dataset-specific work item to a user-facing explanation.

**Information That Should Be Removed or De-emphasized**
- The raw `Current work` string should not be the main explanation.
- The dataset name can be secondary unless it changes the recommendation.
- The debug command should be visually subordinate to the human-readable diagnosis.
- The hedge `if this is a large dataset profile` is too abstract for the first answer.

**Suggested UI / Output Changes**
- Lead with a one-sentence diagnosis: “The run is active and still booting, so it does not look failed.”
- Translate each lifecycle state into plain language.
- Show a clear primary recommendation first, then a secondary “if this persists, debug” action.
- Replace raw internal work strings with a user-facing summary and optionally tuck the raw text under a details toggle.
- If the run is fresh, explicitly say “not stuck yet” or “still within normal startup window.”
- Add a small status taxonomy: queued, booting, running, reconciling, failed, complete, with one-line explanations.
