   - A normal user can see there is an active run, but cannot tell whether waiting is sensible or whether the run is stalled.
   - This affects the decision between waiting and inspecting.
2. The suggested recovery path is actionable but not especially ergonomic.
   - `research debug run 47ff6747-fdc4-4c97-9a2d-72d038de90c6` is useful, but in a terminal it is still just text the user must copy and run.
   - There is no visual emphasis on this as the primary next action.
3. The same conflict is stated three times in slightly different forms.
   - `Blocked: ... busy`, `Starting a duplicate analysis is not allowed...`, and `Next: wait...` all communicate the same core fact.
   - That is not wrong, but it makes the screen feel more verbose than necessary for a busy-state.
4. The layout is visually flat.

   - The user has to read linearly to find the key action.

**Missing Information That Would Have Helped**
- Whether the blocking run belongs to the current user.
- Whether the blocker is expected to clear soon.
- Whether the active run is simply starting normally or is in a problematic state.
- Whether canceling the active run is safe or recommended for this dataset.
- A clearer explanation of what `booting` means in this context.

**Information to Remove or De-emphasize**
- De-emphasize one of the repeated “duplicate analysis” explanations.
- Shorten or visually de-emphasize the raw UUID so the blocking run label carries more weight than the identifier itself.
- The prose `Check it:` line could be replaced with a more structured action list.

**Suggested UI / Output Changes**
- Use a compact conflict block:
  - One headline: `Dataset busy`
  - One cause line: `enriched-tweets is locked by run <id>`
  - One status line: `Current run: booting`
  - One action list: `1. Wait  2. Inspect  3. Cancel`
- Add a short interpretation of lifecycle state, such as what `booting` implies.
- Make the inspect/debug command the primary action, ideally with clearer emphasis than surrounding text.
- If possible, present the next step as a numbered action rather than embedded prose.
- Consider a stronger visual hierarchy so the blocker and next action are immediately scannable.
