**Missing Information That Would Have Helped**
- A short plain-English statement that the remote analysis had started and was now running in the background.
- A lifecycle hint such as “I’ll queue this remotely and you can follow progress in the dashboard.”
- A clearer expectation of what artifact types will appear at completion, since the prompt asked for a bar chart and representative examples.
- A progress cue during the 15-25 second quiet period, even if only a heartbeat or stage label.

**Information That Should Be Removed or De-Emphasized**
- Internal tool names like `start_research_run`.
- Duplicate completion lines that repeat the same queued-run message and URLs.
- Overly verbose raw URLs unless they are the primary action the user needs next.

**Suggested UI/Output Changes**
- Replace internal tool labels with user-facing stages like `Checking datasets`, `Starting remote analysis`, and `Run queued`.
- During long remote work, keep a concise progress line alive so the screen does not appear frozen.
- Add one final sentence after the run id that explains the next step, for example: “The run is queued and will populate results in the dashboard.”
- If a run is expected to produce chart/example artifacts, surface that explicitly in the completion message.
- Consider truncating or collapsing dashboard/terminal URLs to reduce wrap noise in the terminal.
- If the CLI is intentionally hands-off, show that clearly so silence reads as waiting, not stalling.
